# M2 验收标准

> 阅读优先级：**必读**
> 给谁看：Sam（产品负责人）+ Claude Code（开发）+ 远程审计 Claude
> 什么时候用：M2 启动时全员理解，M2 完成时按此验收
> 这份文档决定 M2 算不算"做完"

---

## 0. 为什么需要这份文档

### M1 的教训

M1 验收没有严格标准 → 9 个核心问题留到 M2 才发现：
- 画像数据完全断链
- 老白人格被暴力截断
- 5 个 prompt 是死代码
- 长期记忆 100 条门槛太高
- 等等

**M2 必须吸取教训：上线前严格验收，不留隐患**。

### M2 验收的特殊性

```
开发在另一个系统(Claude Code 或本地)
对话在当前系统(Claude)
我(远程审计 Claude)看不到代码
我只能通过你(Sam)给的材料判断 M2 是否真的做对了
```

所以这份验收标准必须满足 4 个条件：

1. **客观可验证**：不是"感觉做完了"，是"具体某个东西能不能跑"
2. **用户视角**：不是"代码写了"，是"用户能感受到什么"
3. **Sam 能独立验证**：不需要审计 Claude 来跑，Sam + Claude Code 能完成
4. **远程可审计**：Sam 给的材料让审计 Claude 能判断真实性

---

## 1. 4 层验收架构

```
第 1 层:能力清单验收(做了什么)
        ↓
第 2 层:链路图验收(链路是否通)
        ↓
第 3 层:指标验收(用户感受是否变好)
        ↓
第 4 层:M2 完成报告(综合材料,提交审计)
```

每层都必须通过，缺一层 = M2 不算完成。

### 2.1 spec-m2-001：把画像数据接进主对话

**验收点 A1.1：conversation-turn 拼接画像数据**

通过条件：
- conversation-turn.orchestrator 拼的 user message 必须包含：
  - profile_assertions（精炼画像）
  - relationship_observations（老白观察）
  - user_language_fingerprint（用户语气指纹）
  - relationship 表的 stage / 关系类型 / 认识多久

证据材料（Sam 提供给审计 Claude）：

```
1. 修改前后对比
   - 让 Claude Code 跑:
     "对比 M1 末和 M2 完成时,conversation-turn 拼的 
      user message 模板有什么不同。
      把两个版本的模板都贴出来。"
   - 把 Claude Code 的回答转给审计

2. grep 验证
   - 让 Claude Code 跑:
     "grep 'profile_assertion' 在 conversation-turn.orchestrator.ts 
      grep 'profile_assertion' 在 conversation route 文件
      把命中行数贴出来"
   - 应该都有命中

3. 真实对话证据
   - 找一个数据库里有 5+ profile_assertions 的真实用户
   - 让该用户(或 Sam 自己)发一条消息
   - 截图老白的回复
   - 老白的回复必须显式引用画像数据
     例: "她平时不这样回你"
         "你之前说过她在意 X"
         "根据你和她相处这几周,她应该是 Y 类型"
```

**验收点 A1.2：default prompt 加占位符**

通过条件：
- 5 个 default prompt 模板（parsing/reflecting/diagnosing/planning/drafting）
  + conversation-turn 系统 prompt
  必须有画像数据占位符

证据材料：

```
1. 截图 default-prompts.ts 中 prompt 模板
2. 必须能看到类似:
   "[她的稳定特征:{profile_assertions}]"
   "[最近的观察:{recent_observations}]"
   "[兄弟的语气:{language_fingerprint}]"
```

### 2.2 spec-m2-002：修复人格注入 + 加当下情绪识别

**验收点 A2.1：删除 slice(0, 1500) 暴力截断**

通过条件：
- conversation-turn 不再用 `loadPrompt('parsing').slice(0, 1500)`
- 改为独立的人格模块（如 `apps/api/src/ai/laoke-persona-loader.ts`）

证据材料：

```
1. grep 验证
   - 让 Claude Code 跑:
     "grep 'slice(0, 1500)' 整个 apps/api/src/
      grep 'loadPrompt(\"parsing\")' 在 conversation-turn"
   - 都应该 0 命中

2. 新模块存在性
   - 让 Claude Code 列出新增的人格模块文件
   - 必须有完整的人格定义(32 岁兄长 + 不说/常说 + 判断风格)
   - 不超过 800 字
```

**验收点 A2.2：5 个 scene 都注入完整人格**

通过条件：
- parsing / reflecting / diagnosing / planning / drafting / conversation-turn
  都使用同一份人格模块
- cache miss 时人格仍稳定

证据材料：

```
1. 让 Claude Code 跑:
   "grep 'laoke-persona' 或类似关键词 在 6 个 scene 的 prompt 加载逻辑中
    确认 6 处都引用了同一个人格模块"

2. 人格通过率测试
   - 跑 persona-check 测试集
   - 6 个场景的通过率必须都 >= 95%
   - 提供测试结果截图

3. cache miss 测试
   - 模拟 5 个 cache miss 场景
   - 老白回复必须仍像 32 岁兄长
   - 提供 5 个 case 的输出
```

**验收点 A2.3：当下情绪识别能力**

通过条件：
- conversation-turn 的 system prompt 必须包含"当下情绪识别"指令段
- 老白在用户输入有"反常"信号时，必须显式指出反常 + 推断原因

证据材料：

```
1. prompt 内容
   - 截图新版 conversation-turn system prompt
   - 必须包含类似指令:
     "在回复前,先判断:
      她平时是什么样的(看长期画像)
      她今天是否反常(看当下信号)
      如果反常,可能因为什么"

2. 测试用例(10 个反常场景)
   准备如下测试场景,看老白能否识别:
   
   场景 1:她平时秒回长句,今天回个"嗯"
   场景 2:她平时不说脏话,今天骂了一句
   场景 3:她平时主动找话题,今天 3 天没主动
   场景 4:她平时喜欢用表情,今天纯文字
   场景 5:她平时回答详细,今天敷衍
   ...
   
   老白的回复必须显式指出:
   - 反常的具体表现
   - 推断的可能原因
   
   至少 7/10 命中

3. 真实使用证据
   - Sam 自己作为用户测 3 个反常场景
   - 截图老白的回复
   - 标注"老白识别出反常的部分"
```

### 2.3 spec-m2-003：长期记忆门槛降低 + 缓存

**验收点 A3.1：触发门槛可调且降低**

通过条件：
- 默认值从 100 降到 30-50
- 可在 admin 后台 settings 调整

证据材料：

```
1. SystemConfig 表证据
   - 让 Claude Code 查 SystemConfig 表
   - 应该有 long_term_memory_threshold 字段
   - 默认值 30-50

2. admin 后台截图
   - settings 页能看到该参数
   - 改了之后能立即生效

3. 实际触发证据
   - 找一个对话 50 条左右的关系
   - 让老白回复
   - AiCallLog 中能看到 long-term-memory orchestrator 被调用
```

**验收点 A3.2：增量摘要缓存**

通过条件：
- 不再每轮重算
- 只对"新滚出窗口"的部分增量摘要

证据材料：

```
1. grep 验证
   - 让 Claude Code 跑:
     "grep 'cache' 在 long-term-memory.orchestrator
      grep 'incremental' 或 '增量' 在该文件"
   - 应该有命中

2. 性能对比
   - 同一段对话(150 条)
   - M1 状态: 每轮调 Haiku 一次
   - M2 状态: 每 N 条新消息才调一次
   - 提供 7 天对话的 Haiku 调用次数对比
   - M2 应该减少 70%+

3. 延迟对比
   - 老白回复延迟 P50 / P95
   - M2 应该降低 50% 以上(对话 > 100 条的关系)
```

## 3. 第 2 层：链路图验收

针对每个 M2 spec 的核心功能，提供"端到端链路图"+"每个节点的验证证据"。

### 3.1 链路图模板

```
[功能名] 链路图

[节点 1] 用户在 mobile 做什么动作
  验证: ___ (截图)

[节点 2] mobile 发什么 API 请求
  验证: ___ (Network 面板截图)

[节点 3] 后端 route 接收
  验证: ___ (路由文件代码片段截图)

[节点 4] route 查什么数据
  验证: ___ (Claude Code grep 结果)

[节点 5] route 把数据传给 orchestrator
  验证: ___ (orchestrator 函数签名截图)

[节点 6] orchestrator 拼什么 prompt
  验证: ___ (拼接后的 user message 截图)

[节点 7] 送给 AI(Sonnet/Haiku/Gemini)
  验证: ___ (AiCallLog 记录截图)

[节点 8] AI 返回什么
  验证: ___ (回复内容截图)

[节点 9] 用户看到什么
  验证: ___ (用户端截图,标注关键部分)

每个节点都有证据 → 链路通
任何一个节点缺证据 → 链路断
```

### 3.2 必须画的链路图清单

M2 完成时必须提供以下 6 张链路图：

```
链路图 1:用户发消息 → 看到老白懂她(spec-m2-001)
链路图 2:用户在反常情绪场景 → 老白识别(spec-m2-002)
链路图 3:用户对话超过 30 条 → 长期记忆触发(spec-m2-003)
链路图 4:Sam admin 改数据流配置 → 用户端老白行为变化(spec-m2-004)
链路图 5:Sam admin 删除画像 → 用户端老白回复变化(spec-m2-005)
链路图 6:历史维度: 老白引用时间 → 真实可见(spec-m2-006)
```

### 3.3 链路图的标准

每张链路图必须：
- 至少 8 个节点（从用户动作到用户感受）
- 每个节点有具体证据（截图 / grep / 代码片段）
- 证据必须可验证（不是"做了" 是"看到了"）

## 4. 第 3 层：指标验收

不只是"做了"，还要"用户真的感受到了"。

### 4.1 主指标（必须达到）

#### 指标 1：老白引用画像数据的频次

```
测量方法:
  - M2 完成后,1 周内随机抽 50 条老白回复
  - 标注: 是否显式引用画像数据
  - 引用包括: 提到她的特征/习惯/平时风格/反常等

目标:
  >= 60% 回复显式引用画像

提供材料:
  - 50 条样本 + 标注表(可用 admin 后台 annotation 功能)
  - 引用率统计
```

#### 指标 2：用户对老白"懂他"的反馈

```
测量方法:
  - M2 完成前 7 天 vs 完成后 7 天
  - 用户主动 like 中含"老白懂我"类表达
  - 关键词: "懂"/"准"/"看出来"/"记得"/"知道我"等

目标:
  - 占 like 总数比例增加 50%+

提供材料:
  - 两个时间段的 like 内容统计
  - 关键词命中率对比
```

#### 指标 3：老白"反常识别率"

```
测量方法:
  - 抽 20 个用户对话有"反常"信号(平时 X 今天 Y)
  - 看老白回复是否识别反常并指出
  
目标:
  >= 60% 命中

提供材料:
  - 20 个对话样本(脱敏)
  - 老白回复 + 是否识别标注
```

#### 指标 4：用户连续使用率

```
测量方法:
  - 周留存(本周用过 + 下周也用过)
  - M2 完成前 4 周 vs 完成后 4 周

目标:
  - 周留存提升 20%+

提供材料:
  - admin 后台 KPI 页截图
  - 两个时段的对比
```

#### 指标 5：老白人格通过率

```
测量方法:
  - persona-check 自动检测每次回复
  - 统计通过率

目标:
  - 6 个场景(parsing/reflecting/diagnosing/planning/drafting/
    conversation-turn) 都 >= 95%

提供材料:
  - admin LLM 监控页截图
  - 各场景通过率
```

### 4.2 副指标（值得看）

#### 指标 6：LLM 调用成本

```
测量:
  - 长期记忆缓存后 Haiku 调用次数
  
目标:
  - 减少 70%+(对话 > 100 条的关系)
```

#### 指标 7：admin 操作频次

```
测量:
  - 数据流配置面板每周打开次数
  - 画像数据管理页每周打开次数
  
目标:
  - 反映 Sam 是否真的在用
  - 至少每周 1 次以上
```

#### 指标 8：画像数据修改频次

```
测量:
  - admin 后台修改 profile_assertions 次数
  - admin 后台删除 observations 次数
  
目标:
  - 至少有真实修改记录
  - 反映"画像有错时能修"是否真的发生
```

## 6. 审计流程（远程审计 Claude 收到完成报告后做什么）

### 6.1 审计步骤

```
Step 1: 检查证据齐全度
  - 主报告: 有
  - 6 张链路图: 全
  - 各 spec 验收点证据: 全
  - 指标实测: 全
  - 真实使用截图: 有
  - 心虚标注更新: 有
  
  缺一项 → 退回补齐

Step 2: 验证每个 spec 的验收点
  - 按本文档第 2 节的验收点逐项检查
  - 不通过的标记"未完成"
  - 通过率 < 80% → M2 不通过

Step 3: 验证链路图
  - 每张链路图的每个节点是否有证据
  - 证据是否真实可信
  - 任何节点断 → 该功能等于没做

Step 4: 验证指标
  - 主指标 5 个: 必须都达标
  - 副指标 3 个: 看趋势
  - 主指标任何一个未达 → 必须解释原因

Step 5: 验证心虚标注诚实度
  - 是否有应该标但没标的
  - M2 期间新应急方案是否标记
  - 不诚实 → 严重问题

Step 6: 出 M2 验收报告
  - 整体评价: 通过 / 部分通过 / 不通过
  - 哪些功能真的能 work
  - 哪些是"做了但没接通"
  - M3 必须修的
  - M3 可以延后的
```

### 6.2 审计可能的判断

**完全通过**：
- 所有验收点 ✅
- 所有链路图通
- 所有主指标达标
- 心虚标注诚实

**部分通过**：
- 大部分 ✅
- 个别 ⚠️ 但有合理解释
- 主指标 4/5 达标
- 出"部分通过"报告

**不通过**：
- 多项 ❌
- 链路图有断点
- 主指标多项未达
- 出"M2 不通过,必须修复后重审"报告

## 7. 验收的几个关键原则

### 7.1 不接受"代码合并就算完"

M1 的错就是这样犯的。
M2 必须"用户视角验证"。

### 7.2 不接受"我们尽力了"

M2 是 14-16 周的事，不是赶工。
该达标必须达标。
没达标必须解释 + 给修复计划。

### 7.3 不接受"指标没法测"

任何能上线的功能，都能测。
说"没法测"= 没真的做。

### 7.4 接受"诚实标注的不完美"

M2 不可能 100% 完美。
**诚实标注 + 修复计划** > 假装完美。

诚实是底线。
