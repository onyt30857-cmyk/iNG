## 1. M2 是什么

### 1.1 一句话

**M2 不是加新功能，是把 M1 已有的东西接通，让产品真正实现新定位。**

### 1.2 新定位（M1 后期才确定的）

> **练爱：你和她每一次沟通的陪练。**
> **从你认识她那一刻开始，到你们关系里的每一次互动。**
> **AI 老白懂这个具体的她（长期画像 + 当下情绪），陪你把每一次沟通做得更对。**
> **用得越久，老白和你都越懂她。关系也因此越走越稳。**

### 1.3 M2 必须解决的核心矛盾

```
M1 数据生产端做得很好:
  ✅ profile_assertions(精炼画像)有了
  ✅ relationship_observations(老白观察)有了
  ✅ user_language_fingerprint(用户语气指纹)有了
  ✅ 长期记忆 orchestrator 有了
  ✅ admin 后台 80-85% 完成

M1 数据消费端基本断了:
  ❌ 主对话 conversation-turn 看不到画像数据
  ❌ 老白人格被 slice(0, 1500) 暴力截断注入
  ❌ 关系信息只取了名字,不知道阶段
  ❌ 没有"当下情绪"识别能力
  ❌ admin 后台无法控制"老白看什么"

简单说:
  数据库有满满的"懂她"资产
  但老白主对话时看不到
  用户感受不到"AI 真懂他和她"
```

**M2 的全部工作就是把这个核心矛盾解掉。**

## 2. M2 的 3 大主轴

### 主轴 A：把"懂她"接通（产品质量提升）

让老白真的能看到画像数据、识别当下情绪、人格稳定。

包含 3 个 spec：
- **spec-m2-001**：把画像数据接进主对话
- **spec-m2-002**：修复人格注入 + 加当下情绪识别
- **spec-m2-003**：长期记忆门槛降低 + 缓存

### 主轴 B：AI 输入控制后台（让产品负责人能调）

补上 M1 后台唯一大缺口——AI 输入控制类。

包含 2 个 spec：
- **spec-m2-004**：数据流配置面板
- **spec-m2-005**：画像数据管理后台

### 主轴 C：工程债清理（基础健康）

清理 M1 留下的应急方案和死代码。

包含 1 个 spec：
- **spec-m2-006**：工程债清理（死代码 + 应急方案 + 历史维度）

## 3. M2 不做什么

明确**不做**的清单：

### 3.1 不加新功能

不做：
- ❌ 5 模式增强（兄长 / 镜照 / 陪伴 / 关怀 / 承认 / 推一把）
- ❌ 老白信
- ❌ 季度档案
- ❌ 数据导出
- ❌ Session Chain（衔接下一步）
- ❌ 视觉 OCR 大改（M1 已用 Claude Sonnet 4 vision，足够）

理由：这些都基于"老白能看到画像"假设。M2 把假设变成现实后，M3 再考虑这些。

### 3.2 不做推广

不做：
- ❌ 校园冷启动
- ❌ 工厂男性场景
- ❌ 裂变机制
- ❌ KOL 后台
- ❌ 学生认证

理由：产品质量没上来前推广 = 烧钱不出效果。M2 完成后产品质量上来，M3 再考虑推广。

### 3.3 不做业务运营

不做：
- ❌ 优惠券系统
- ❌ 邀请奖励
- ❌ 推广归因
- ❌ 付费转化漏斗优化

理由：当前付费用户量级未到，M3+ 再做。

### 3.4 不再写大开发包

不做：
- ❌ 18 份大开发包式的文档
- ❌ 51 份累积式文档

M2 总文档量 8 份，精简但完整。

## 5. M2 必须遵守的 4 条指导原则

这 4 条原则**基于 M1 真实犯过的错总结**，是 M2 的护栏。
所有 spec 必须遵守。

### 原则 1：交付的不是"代码做完了"，是"用户真的拿到了"

每个 spec 必须有"验收路径"：

```
用户做什么动作 → 在产品里能感受到这个改进
```

如果答不出"用户怎么感受到"，spec 等于没做。

**M1 的错**：
- spec-008 抽画像数据 → 代码做完了 → 老白看不到 → 用户感受不到
- 5 个精细 prompt → 代码做完了 → 生产几乎不调用 → 死代码

### 原则 2：把"应急方案"和"正式方案"分开标记

任何应急、临时、降级的实现，必须**显式标记**：

```typescript
// HACK: 临时实现,2026-Q3 必须替换
//       原因: ...
//       影响: ...
//       何时修: ...
```

或在文档里：

```
心虚标注 #X:
- 状态: 应急方案,M3 必须改
- 原因: ...
- 影响: ...
- 何时修: ...
```

**M1 的错**：
- `slice(0, 1500)` 应急方案变永久方案
- 长期记忆每轮重算变永久方案

### 原则 3：每个新功能必须先画"端到端链路图"

每个 spec 必须包含链路图：

```
用户做什么动作 →
  前端发什么请求 →
    后端 route 查什么数据 →
      orchestrator 拼成什么 prompt →
        AI 收到什么 →
          AI 回什么 →
            用户看到什么
```

链路图的**每个节点**必须有责任人 + 验证方式。

**M1 的错**：
- 各个 spec 单独做的都对
- 但没人画过完整链路图
- 结果链路上有断点没人发现

### 原则 4：分段做事，但每个里程碑必须验证全链路

每 2-3 周一次"全链路体检"：

```
体检日不写新代码:
  - Sam 真实使用产品 1 小时
  - 列出"我以为应该有但没有"的瞬间
  - 用 Claude Code 查每个瞬间背后的链路
  - 找出断点
  - 修复后才能开下一个 spec
```

**M1 的错**：
- 12 周开发 M1
- 没有一个里程碑专门验证"所有链路连通"
- 上线时多处断点

## 8. M2 风险预案

### 风险 1：实施过程中发现新问题

预案：
- 按原则 4，每 2-3 周全链路体检
- 发现新问题立即评估优先级
- 优先级高于既定 spec → 调整顺序
- 优先级低 → 进 M3 待办

### 风险 2：spec 实施超期

预案：
- 单 spec 超期 1 周以内 → 继续
- 超期 2 周以上 → 拆分 spec，部分进 M3
- 不接受"为了赶工降低验收标准"

### 风险 3：用户感受不到改进

预案：
- 按原则 1，每个 spec 完成后做"用户视角验证"
- 用户感受不到 → 链路有断点 → 找出补齐
- 不能"代码做完了就算"

### 风险 4：M2 完成后真实用户反馈不达预期

预案：
- 不立即推广
- 找 5-10 个真实用户深度访谈
- 收集"还缺什么"
- 进 M3 规划

### 风险 5：Claude Code 实施出现重大偏差

预案：
- 启动 M2 时把这 8 份 PRD 全部传给 Claude Code
- 实施过程中每个 spec 完成后，Sam 先粗看一遍
- 有偏差立即纠正，不积累
