# 12 周开发指南

> Sam 一人 + Claude 全程开发,12 周从 0 到 M1 上线。

## 这份指南怎么用

这份指南是**给 Sam 看的**,不是给 Claude 看的。

- 每周开始时读对应章节,确认本周目标
- 每天结束时回看进度,确保没掉队
- 遇到问题时翻第 13-14 节"常见坑"

详细完整版见 `练爱_Claude全程开发指南_v1.0.docx`(Word 文档,带视觉)。

## 1. 你的开发模式

整个项目的开发**全部由 Claude 写代码**。你做的事:
- 写 spec(需求规格)
- review Claude 写的代码
- 真机测试
- 决策技术方向
- 维护项目宪法(CLAUDE.md)

这意味着:你需要每天 6-8 小时在电脑前。

## 3. Week 1:脚手架(最关键)

第 1 周是后续 11 周的基础。这周质量决定后续效率。

### 第 1-2 天
- 申请所有账号:阿里云(OSS、内容安全)、Apple Developer、Anthropic API、Gemini API
- 建立 GitHub 仓库,把 lianai-dev-kit 解压进 `docs/` 目录
- 配 CLAUDE.md(已在 lianai-dev-kit 里,直接复制到根目录)
- 初始化 monorepo:用 pnpm workspace
- 跑 spec-001 的 Tasks(让 Claude 列 tasks,你确认)

### 第 3-5 天
- 后端 Fastify + Prisma + PostgreSQL 初始化
- 前端 uni-app x 项目初始化
- AI client 统一封装(ai/client.ts)
- 一个 Hello World 接口跑通(从前端发起,后端返回,Claude API 调通)

### 第 6-7 天
- iOS 真机测试(配 Apple Developer 证书)
- 安卓真机测试(adb 调试)
- H5 本地预览
- 写 README,记录所有"能跑"的步骤

**Week 1 验收**:第 7 天结束前,你能在 iPhone 上看到一个空白的"练爱" App 启动起来,且能调通后端的 hello world 接口(后端会调 Claude API 返回一句"老 K 在听")。

## 4. Week 2-3:登录 + 关系档案

## 5. Week 4-5:核心复盘(产品的灵魂)

这是产品最重要的 2 周。所有产品质量都建立在这一段。

### Week 4:OCR + PARSING(spec-004 + spec-005 部分)
- 阿里云 OSS 接入(直传凭证)
- Gemini 2.5 Flash 接入(OCR)
- 30 张测试集准确率达标
- 复盘状态机 ENTRY → PARSING 跑通
- PARSING 的 prompt(`03-prompts/parsing.md`)接入

### Week 5:剩下的状态(spec-005 完成)
- REFLECTING、DIAGNOSING、PLANNING、DRAFTING 全部跑通
- SSE 流式输出
- DRAFTING 的 3 条话术(产品价值的"成品")
- 危机干预独立流程
- 30 个测试 case 全部通过

**Week 5 验收**:你能完整跑一次"上传 5 张冷处理截图 → AI 帮你看 → 给三方向话术"的流程,且话术看起来像老 K 写的(不是 ChatGPT)。

## 9. Week 9-10:内测

找 30 个真实用户内测。每天根据反馈改产品。

### 种子用户来源
- 小红书发帖招募(自己写真诚的招募帖)
- 朋友介绍(朋友的朋友)
- 知乎相关问题下评论引导

### 内测准备
- TestFlight (iOS) + 应用市场内测包(安卓)
- 反馈渠道:用户群 + 用户访谈
- 准备一份"内测须知":用户预期、隐私说明、反馈方式

### 工作节奏
- 上午:看用户反馈、列 bug 清单
- 下午:让 Claude 改 bug、跑测试
- 晚上:发新版本给用户

## 10. Week 11:上架

### 应用市场清单
- 苹果 App Store(中国区)
- 华为应用市场
- 小米应用商店
- OPPO 应用商店
- vivo 应用商店

### 应用市场常见拒绝理由
- 涉及恋爱话术的描述太直接
- 截图素材有"撩""PUA"等敏感词
- 隐私政策不完整
- 内购流程不规范

### 处理建议
- 产品描述用中性词:"AI 沟通辅助""高情商表达助手"
- 截图避开过于直接的"撩"字眼
- 隐私政策必须明确数据如何使用
- 实在搞不定:找上架代办

## 13. 常见坑和处理

### 坑 1:Claude 调用了不存在的函数/库
处理:
- 让 Claude 跑代码,看能不能跑通
- 跑不通时让它先 web search 确认 API 是否存在
- 必要时直接给它正确的 API 文档链接

### 坑 2:Claude 改了不该改的文件
处理:
- 严格 review git diff,有越界立刻撤销
- spec 里明确"不准改 X 文件"
- 用 git 分支隔离

### 坑 3:Claude 写的代码看似对但实际没跑过
处理:
- 让 Claude 必须真跑一遍才声明完成
- 命令:"你的代码跑通了吗?给我看运行结果"

### 坑 4:长上下文丢失
处理:
- 每个新 spec 开新会话
- 每周更新 CLAUDE.md
- 长会话主动 /compact

### 坑 5:iOS 真机部署
- Apple Developer 证书过期会突然不能装新版本
- TestFlight 审核需要 1-2 天
- 某些 API 在模拟器正常但真机不正常

### 坑 6:安卓 ROM 兼容
- 华为某些权限默认拒绝
- 小米的"神隐模式"会杀后台
- OPPO/vivo 的通知系统差异

## 14. 心虚清单(必须诚实告诉 Sam)

### 14.1 你的产品 M1 会有"瑕疵"
代码质量、性能、边界处理——会有 Claude 没考虑到的坑。M1 上线后会有 bug,你会持续修。这是正常的,不是失败。

### 14.2 你会变得"半个工程师"
6 个月后你会懂数据库、懂 API、懂前端基本逻辑——不是因为你想学,是因为你必须学才能做好 review。这是好事,但意味着你的精力分布会被改变。

### 14.3 节奏会快但累
传统团队 4-5 人 6 个月做的事,你一个人 + Claude 12 周做完——意味着你的工作强度比一个普通工程师还大。第一个月你会觉得"靠 Claude 真神",第二个月你会觉得"为什么我天天在 review"。

### 14.4 第一次崩盘是必然的
某一天 Claude 会改坏一个核心功能。你会花一整天时间排查。这是必然的,预算里要算上这种"灾难日"。

### 14.5 Claude 不能完全替代经验
某些"不写在文档里的坑"(iOS 平台特性、安卓 ROM 兼容、应用市场博弈)Claude 能给建议但不一定准。这种时候找有经验的人补位是对的,不是失败。

## 15. 一句话总结

> AI 负责在确定性空间里高速执行,人负责维护和扩展那个确定性空间的边界。

你掌握那个"确定性空间的边界"——产品要做什么、不做什么、做到什么程度。Claude 在你定义的空间里高速执行。这是这套模式的本质。

12 周后,M1 上线。
